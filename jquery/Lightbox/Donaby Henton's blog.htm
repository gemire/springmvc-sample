One can never be too rich, too thin or have too many web service clients

I’ve spruced up some old code for spring-ws from about a year and a half ago to use jetty and pom files. This time I’m in netbeans (for some of these), just to keep my hand in.

Spring WS

  • client code:
    http://code.google.com/p/springmvc-sample/source/browse/trunk/netbean_webservices/springwebservice
  • server code:

  • client code:
    http://code.google.com/p/springmvc-sample/source/browse/trunk/netbean_webservices/cxfclient (eclipse)

Heaven forbid that I do something like that in an object oriented language!  “In my day, sonny, we used to subclass our code, and I walked 10 miles to school uphill both ways!”
Its tough being a dinosaur….

Convention over Common Sense

So you’ve heard about it, all these new fangled frameworks and their talk about ‘convention over configuration’.  Its all over Spring (ever since VMWare bought them out) and Maven. The idea is that reasonable assumptions can be made about how something should be set up and that defaults will be filled in and off you go.

In spring, this is things like ‘annotation-driven’ for MVC and in Maven its things like surefire looking for classes ending in *Test or TestCase.

What kind of nonsense is this?   This is a marketing gimmick, pure and simple. Somebody who knows more than you sets things up the way he likes it and then doesn’t tell you. What a nice way to create the illusion  of superiority. I know the secret handshake and you don’t, and by the way, if you pay for my junket to your conference, I’ll let you in on it.

Grrrh……..

05Aug

Stupid Maven Tricks

Uncle Don has a problem. He gets stuck having to wade his way through a huge code base which uses 30-40 in house jar files for various classes and due to circumstances beyond everyone’s control can’t find the jar to include for a given class. The maven POM refers to these guys, but the question is which jar file contains the class of interest. So here’s what  your Uncle Don does, the bozo that he is:

  1. create a fake project
  2. create a fake class
  3. place an import statement in the fake class for the class you are trying to locate
  4. click on the eclipse lightbulb thingy that indicates the import can’t be resolved
  5. select ‘fix imports’
  6. It should find the jars and suggest the different versions

This may be a myeclipse feature only. Also, I bet it knows to search repositories only if they are listed in your .m2/settings.xml file, which means that your in-house stuff had better be there. But hey, it works!

Cool thing to do with log4j

So I learned a cool log4j trick today. You push your user’s login id into the MDC (Mapped Diagnostic Context) of log4j, that way it shows up on any log entry that they make. I stole some code off the internet for a web servlet filter that will do just that. The code is located in my wiki here, along with a url to credit the original author.

Armed with this information and the approximate time of any reported errors, you can check the logs and have a much better chance of finding a log entry. Now finding a useful log entry is a different story…
